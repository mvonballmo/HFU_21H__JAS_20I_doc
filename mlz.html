Bewertungskriterien MLZ

Es soll ein App geben womit man:

  • Daten von einem Server abfragen und anzeigen kann.
  • Die Daten können vom Benutzer geändert und gespeichert werden.
  • Die Daten können auch gelöscht werden.

Bewertung Aspekte

Der App beinhaltet zumindest die folgende Aspekte

  • Daten von mindestens einer Entität sollen vom Server abgeholt und in einer Liste angezeigt werden.
  • Details einer Entität können angezeigt werden (Liste/Detail)
  • Die Daten sollen aus verschiedenen Typen (z.B. Datum, Text, Zahlen, Boolean, usw.) bestehen.
  • Einen Datensatz kann in einem Formular editiert werden, mit Validierung von den verschiedenen Typen. Zumindest ein Feld muss ein Zwangsfeld sein.
  • Die Validierung muss mindestens so ausgeprägt sein, dass Fehler auf der Seite beim Drücken des Speichern-Buttons angezeigt werden. Dies darf nicht über einen alert geschehen, sondern die Infos müssen per JavaScript dem DOM hinzugefügt werden.
  • Ist das Formular fehlerfrei und der Benutzer klickt auf den Speichern-Button soll ein JSON Objekt mit allen eingegebenen Informationen anzeigt an dem lokalen Server gesendet werden.
  • Ein neues Objekt kann erstellt werden.

Zusatzpunkte gibt's für:

Ausgereifte Validierung

  • Die benötigten Felder müssen vorgängig validiert werden und der Button zum Abschicken des Kontaktformulars soll nur aktiv sein, falls keine Fehler mehr vorhanden sind.
  • Fehler sollen direkt dort angezeigt werden wo sie entstehen (beispielsweise mit einem roten Rahmen und einem Informationstext über dem Problem).
  • Einschränkung der Länge von Text-Feldern validieren (z.b. eine Maximallänge)

Suchen

  • Der Benutzer kann nach Freitext in den Feldern suchen, um passenden Entitäten zu finden. Die gefundene Entitäten werden in einer Liste angezeigt.
  • Der Benutzer kann einschränken, welche Felder durchgesucht werden.
  • Die gefundene Texte aus den Entitäten werden angezeigt.
  • Der Suchtext wird in den gefundenen Texten hervorgehoben (highlighted)

Benutzerkomfort

  • Gewisse Felder mit localStorage im Browser des Users speichern, so dass. Wenn der User die Seite wieder aufruft, sollen die Felder mit den gespeicherten Werten ausgefüllt werden.
  • Der Benutzer kann Test-Daten erstellen lassen.
  • Der Benutzer kann allen Daten löschen.

Multi-Entität

  • Die App hat mehrere miteinander verbundene Objekttypen (z.B. Firmen und Personen und Herkunftsländer). In dem einen Formular kann Entitäten aus einer anderen Listen selektiert werden (z.B. bei einer Person die Firma selektieren).

Mandantenfähigkeit/Mehrere Benutzer

  • Der Benutzer kann sich anmelden und nur die eigenen Daten anzeigen lassen.
  • Die Anmeldung ist mit einem Passwort gesichert.

Allgemeine Bewertung

Die Software hat eine ersichtliche Architektur

  • Ordner und Dateistruktur ist vorhanden und logisch
  • Die erlernten Pattern wurden angewendet

Die Benutzeroberfläche erfüllt ergonomische Grundregeln

  • Die Navigation ist logisch und konsistent
  • Das Styling wurde konsequent umgesetzt
  • Die Bedienung der App fühlt sich natürlich an

Die Implementation ist strukturiert

  • Es findet sich kein auskommentierter Code
  • Es wurde Code wo nötig und sinnvoll ausgelagert
  • Services wurden sinnvoll erstellt und eingesetzt

Die Applikation ist lauffähig und hat keine grösseren "Bugs"

  • Die App kann nicht sofort zum Absturz gebracht werden
  • Zu erwartende Exceptions sind gehandelt

Notwendige Code-Stellen sind dokumentiert

  • Öffentliche und nicht einfach verständliche Code-Stellen sind dokumentiert

Die richtige Funktionsweise ist mit Tests verifiziert

  • Es gibt Unit-Tests für die zentralen Punkte der App
  • Es wurde manuell getestet und dokumentiert

  • Offensichtliche Fehler?
  • Bad Practices?
  • Eigene Ideen?

  • Sinnvolle Variablennamen?

Präsentation

  • Die Präsentation ist sachlogisch aufgebaut.
